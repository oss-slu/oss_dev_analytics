## Development Workflow
* Branching Strategy: Each GitHub issue must have its own dedicated branch.
* Branch Naming: Use the format feature/issue-number-description or bugfix/issue-number-description.
* Pull Request (PR) Submissions: All PRs for the current sprint cycle are expected to be submitted by **11:59 PM every Friday**
* Templating: All PR's and Issues must follow the provided templates unless otherwise discussed

## Code Review & Merging
* Merging Schedule: Contributors can expect submitted PRs to be reviewed and merged by the following Monday.
* Communication Loop: PR comments and requested changes must be responded to within 24 hours, unless a team member has previously communicated a reason for their unavailability.
* Review Quality: Reviews should focus on both code functionality and adherence to the project's GUIDELINES.md.

## Meeting Schedule & Availability
* In-Class Meetings: The team will meet every Monday during scheduled class time.
* Absences: If a member cannot attend a meeting or meet a Friday PR deadline, they must notify the team via the Slack channel as early as possible.



## Communication Standards
* Primary Channel: All technical and logistical coordination occurs in the #oss-dev-analytics Slack channel.
* All memebers are expected to respond to the team Slack within 24 hours unless there is a break or other extenuating circumstances
* Transparency: Team members are expected to update the Kanban board with active issues they are working on (can also update Slack) to avoid duplication of effort.
